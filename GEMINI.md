# GEMINI.md (fuga言語 開発ガイドライン)

本書は、fuga言語の処理系（インタプリタおよびコンパイラ）開発プロジェクトにおけるビルド、テスト、および設計規約を定めたものである。

---

## 1. プロジェクト概要

本プロジェクトは、特異なメモリ消滅規則（一度参照された変数は消滅する）を持つ静的型付け言語「fuga言語」のインタプリタ（実行環境）およびコンパイラの開発を行う。

*   **言語仕様書:** [fuga-spec.md](file:///home/iimura/f-lang/fuga-spec.md)
*   **ユーザーリファレンス:** [fuga-reference.md](file:///home/iimura/f-lang/fuga-reference.md)

---

## 2. 開発プロセスと注意事項

本プロジェクトでの開発にあたっては、以下の2点を厳守すること。

### 2.1. 計画立案の徹底 (Planning First)
いかなる実装や修正も、**まず計画を立ててから**取り掛かること。
*   アーキテクチャの設計、依存関係、およびfuga言語の「消滅規則」に与える影響を事前に整理し、実装プランを作成・合意した上でコーディングを開始する。

### 2.2. テストの徹底 (Testing)
実装した処理系の動作を保証するため、**テストコードによる検証**を徹底すること。
*   単体テスト（Unit Test）だけでなく、fuga言語のソースコード（`.fuga`）を実際に評価して期待通りの挙動（正常終了、例外送出、消滅エラーでのアボートなど）になるかを検証する統合テスト（Integration Test）を必ず実装・実行する。

## 3. 開発言語・ビルド・テストコマンド

開発言語には、コンパイラ記述とパターンマッチングに優れ、fuga言語の所有権/消滅規則と親和性の高い **Rust** を推奨する。以下はRustを用いた場合の標準コマンドである（他の言語を使用する場合は本章を書き換えること）。

### 3.1. 基本コマンド

*   **ビルド:** `cargo build`
*   **リリースビルド:** `cargo build --release`
*   **テスト実行:** `cargo test`
*   **インタプリタの実行:** `cargo run -- <fugaソースファイル>`
*   **コンパイラの実行:** `cargo run --bin fugac -- <fugaソースファイル> -o <出力パス>`

## 4. 実装上の最重要設計指針

fuga言語の処理系を実装するにあたり、以下の仕様を厳密にシミュレートしなければならない。

### 4.1. ライフサイクル（消滅規則）の追跡
*   **インタプリタ (実行時追跡):**
    *   変数および配列要素の状態（有効/消滅）を追跡するフラグテーブルをメモリ上に維持する。
    *   式評価時、右辺で変数が評価されたら、その評価ステップの直後に即座にフラグを「消滅（無効）」に変更する。
    *   すでに「消滅」フラグが立っている変数にアクセスした場合は、即座に未定義参照エラーとしてアボートする。
*   **コンパイラ (静的データフロー解析):**
    *   静的解析フェーズで変数の生存解析（Liveness Analysis）を行い、同一変数に対する2回目の参照が「消滅後」に発生していないかをコンパイル時に検証する。

### 4.2. 自己代入（自動再定義）の処理
*   `$x = $x + 1.` のような自己代入は、右辺の評価によって一度 `$x` が消滅するが、左辺への代入が成功した時点で「有効」として再定義されなければならない。
*   右辺の評価順序は「左から右」であるため、`$x = $x + $x.` のように右辺で複数回同一変数を評価している場合は、2回目の `$x` の評価時に「消滅済」エラーとなり、自己代入の例外は適用されない。

### 4.3. ダイナミックスコープの変数解決
*   関数呼び出し時、静的なレキシカルクロージャではなく、実行時のコールスタック（呼び出し履歴）を親方向に遡って変数を探索する。
*   **読み取り専用規則:** 探索した結果、現在実行中の関数のローカルフレーム以外（呼び出し元フレームやグローバルフレーム）で発見された変数は、**「読み取り専用」**としてマークされる。
    *   読み取り専用変数は、評価されても**消滅しない**。
    *   読み取り専用変数への書き込み（代入）が試みられた場合はエラーとする。

### 4.4. ループと例外処理による制御フロー
*   `loop { ... }` の制御を抜ける唯一の手段は、例外の送出である。
*   例外が発生した際、コールスタックを順次ポップ（巻き戻し）しながら、最も近い `try-catch` の対応する `catch` 節へ制御をジャンプさせる例外ハンドリング機構を実装する。

## 5. コードスタイルと規約

*   **エラーメッセージ:** 構文エラー（Syntax Error）、型エラー（Type Error）、消滅した変数の参照（Destruction Error）は、発生したソースコード of 行・列番号とともに明確なエラーを出力すること。
*   **テストケースの蓄積:** 各演算子の優先順位（EBNFに基づく）や、変数の消滅挙動を検証するための小さなテストコード（`.fuga`）を多数作成し、自動テストスイートで検証すること。
